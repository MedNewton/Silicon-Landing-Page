# i18n (English + Italian) — Design Spec

Date: 2026-04-25
Status: Approved, ready for implementation plan

## Goal

Localize the entire landing page into Italian and English, with **Italian as the default locale**. Italian uses bare URLs (`/`, `/mobile`, etc.); English is served under the `/en` prefix. A working language switcher in the header lets users jump between locales while preserving the current path.

## Scope

**In scope**
- Install and configure `next-intl` for the App Router
- Move all routes under a `[locale]` segment
- Translate every user-visible string across all sections (header, hero, CTAs, bento, comparison, roadmap, how-it-works, consultants, testimonials, AI-document-generation, pricing, footer — desktop and mobile)
- Provide complete Italian translations for every English string
- Rewire `LanguageSwitcher` to actually navigate between locales (preserving pathname)
- Update middleware to combine next-intl locale detection with the existing mobile-UA rewrite

**Out of scope**
- New languages beyond IT and EN
- Translating the URL `/mobile` segment or any other route name (only the page contents are translated)
- Translating the testimonial *names* (placeholder data — names stay as-is, but their quotes are translated)
- Date/number formatting changes beyond what next-intl offers out of the box
- SEO `hreflang` tags, sitemap localization, robots updates

## Approach

### Library

`next-intl` — the de-facto i18n solution for Next.js 15 App Router. Provides server-component-aware translations (`getTranslations`), a client hook (`useTranslations`), locale-aware routing, message catalogs in JSON, and a middleware factory that integrates with our existing custom middleware.

### Routing structure

`localePrefix: "as-needed"` from next-intl. Italian (default) URLs stay bare; English URLs gain a leading `/en`:

| Locale  | Desktop URL  | Mobile URL       |
|---------|--------------|------------------|
| Italian | `/`          | `/mobile`        |
| English | `/en`        | `/en/mobile`     |

Internally, all four routes resolve to the same `[locale]` page tree:

```
src/app/
├── layout.tsx                    # Root <html> shell, no locale logic
├── [locale]/
│   ├── layout.tsx                # Per-locale layout (loads messages)
│   ├── page.tsx                  # Desktop homepage
│   └── mobile/
│       └── page.tsx              # Mobile homepage
└── (other top-level files)
```

The current `src/app/page.tsx`, `src/app/layout.tsx`, `src/app/mobile/page.tsx` are moved under `[locale]/`.

### Middleware

Order: locale detection runs first via `next-intl/middleware`, then the existing mobile-UA rewrite is applied to the locale-resolved path. The new middleware:

1. Detects mobile/tablet via `userAgent(req)` (existing logic)
2. Resolves locale from URL prefix or cookie (handled by next-intl)
3. If on a homepage path (`/`, `/team` for Italian; `/en`, `/en/team` for English) **and** the device is mobile/tablet, rewrites to the corresponding `/mobile` variant within the same locale
4. Otherwise passes through next-intl's response unchanged

The matcher must include both `/` and `/en/...` paths plus exclude `/api`, `/_next`, static files (next-intl's default matcher handles the static exclusions).

### Message catalogs

Two files at `messages/it.json` and `messages/en.json`. Top-level structure mirrors components:

```jsonc
{
  "header": { /* nav labels + dropdown items */ },
  "hero": { /* badge, title, subtitle, CTA labels */ },
  "comparison": { /* title (rich text), subtitle, card headings, item arrays */ },
  "roadmap": { /* title, subtitle, steps array */ },
  "howItWorks": { /* title, subtitle, steps array */ },
  "consultants": { /* badge, title, subtitle, button */ },
  "testimonials": { /* title, subtitle, topRow array, bottomRow array */ },
  "aiDocumentGeneration": { /* title, subtitle, features array, button */ },
  "pricing": { /* title, toggle, tiers, features array */ },
  "footer": { /* product/company columns, copyright, contact */ },
  "bento": { /* title, card titles, descriptions, chips */ },
  "cta": { /* createProject, watchDemo */ },
  "languageSwitcher": { /* aria-labels (optional) */ }
}
```

I write both files in the same pass. Every key in `en.json` has a corresponding entry in `it.json`. No keys are missing or `null`.

### Rich text and formatting

next-intl's `t.rich()` handles inline markup:

- Hero badge: `"<bold>800</bold> SMEs creates documents daily"` → `t.rich("hero.badge", { bold: (chunks) => <span style={{ fontWeight: 700 }}>{chunks}</span> })`
- Comparison title: `"Creating a project today is <strike>complex</strike>"` → `t.rich(...)` with a `strike` formatter that wraps in the existing strikethrough Box

Hardcoded `<br />` line breaks in headings are removed; CSS `maxWidth` already controls wrapping naturally and translations rarely match line counts.

### Language switcher rewire

`LanguageSwitcher` becomes locale-aware:
- Reads the active locale from `useLocale()` (next-intl)
- Reads the active path from `usePathname()` (next-intl, locale-aware)
- On click: calls `router.replace(pathname, { locale })` from `next-intl/navigation`
- The flag + label shown in the trigger reflect the active locale
- next-intl persists the selection automatically via the `NEXT_LOCALE` cookie when `localePrefix: "as-needed"` is in use

### Component changes

For each component currently containing hardcoded strings:
- Server components (Hero, Bento, Footer, etc.) call `useTranslations` if they're already client components, or `getTranslations` (async) if server. We keep current client/server boundaries unchanged.
- Strings → `t("namespace.key")` calls
- Arrays of items (testimonials, pricing features, comparison items, roadmap steps, how-it-works steps) → `t.raw("namespace.array")` returning the typed object array
- The shared `navItems.tsx` becomes a client-side function that builds nav items from `t()` calls (or runs in a client component context)

### Type safety

Enable `next-intl` typed messages by adding a `global.d.ts` declaration:
```ts
import type messages from "../messages/en.json";
declare global {
  type IntlMessages = typeof messages;
}
```
This makes `t("hero.title")` autocomplete and fail at type-check time for invalid keys.

## Architecture

### File map

**New files**
- `messages/it.json` — Italian translations
- `messages/en.json` — English translations
- `src/i18n/routing.ts` — locale list, default, prefix mode
- `src/i18n/request.ts` — server-side message loader for next-intl
- `src/i18n/navigation.ts` — re-exports `Link`, `useRouter`, `usePathname` configured for the routing
- `src/global.d.ts` — `IntlMessages` type declaration

**Restructured**
- `src/app/page.tsx` → `src/app/[locale]/page.tsx`
- `src/app/layout.tsx` → split: a thin `src/app/layout.tsx` (root `<html>`) + `src/app/[locale]/layout.tsx` (loads messages, applies font + theme)
- `src/app/mobile/page.tsx` → `src/app/[locale]/mobile/page.tsx`

**Rewritten**
- `src/middleware.ts` — combine next-intl middleware with mobile-UA detection
- `src/components/layout/languageSwitcher.tsx` — wire to `next-intl/navigation`
- `src/components/layout/navItems.tsx` — convert to a hook or function that pulls labels from translations
- Every component with hardcoded user-visible strings (full list under "Rollout sequence")

**Modified**
- `next.config.js` — add `next-intl/plugin` wrapper
- `package.json` — add `next-intl`

### Why split into two phases

Phase 1 lays the groundwork without changing any visible content. The site still renders in English at this point because keys haven't been wired to components yet; messages contain both languages. The infrastructure can be verified in isolation (URL routing works, language switcher navigates between locales, no 404s, lint/typecheck pass).

Phase 2 replaces strings section by section. Each commit is small and reversible. Visual verification per section catches missed keys and rendering issues incrementally.

## Rollout sequence

### Phase 1 — Infrastructure (one or two commits)

1. Install `next-intl`
2. Author `messages/it.json` and `messages/en.json` with every key filled in for every section
3. Create `src/i18n/routing.ts`, `src/i18n/request.ts`, `src/i18n/navigation.ts`
4. Add `src/global.d.ts` for typed messages
5. Move `app/page.tsx`, `app/layout.tsx`, `app/mobile/page.tsx` under `app/[locale]/`; split root layout
6. Rewrite `src/middleware.ts` to chain next-intl + mobile detection
7. Update `next.config.js` with `next-intl/plugin`
8. Rewire `LanguageSwitcher` to navigate locales

### Phase 2 — String replacement (one commit per section)

9. `Header` + `MobileHeader` + `HeaderDropdown` + `MobileNavAccordion` + `navItems`
10. `Hero` + `MobileHero` + `CTA` + `MobileCTA`
11. `BentoSection` + `MobileBusinessPlanSlider`
12. `ComparisonSection`
13. `RoadmapSection`
14. `HowItWorksSection`
15. `ConsultantsSection`
16. `TestimonialsSection`
17. `AiDocumentGenerationSection`
18. `Pricing` + `MobilePricing`
19. `Footer` + `MobileFooter`

### Verification (final task)

20. `npm run check` clean
21. Visual checks at `/`, `/en`, `/mobile`, `/en/mobile`
22. Language switcher round-trip (IT → EN → IT) preserves the current scroll position / pathname
23. All section anchor links (`#features`, `#pricing`) still work in both locales

## Risks / known unknowns

- **Italian copy quality.** I write the Italian translations as a first pass; idiomatic phrasing for marketing taglines (e.g. "Turn your idea into a strategic plan") may need a native speaker's polish. Translations live in plain JSON and are easy to edit later.
- **Middleware ordering edge cases.** Next-intl's middleware may rewrite or redirect; chaining it with our mobile-UA logic must preserve both behaviors. The implementation detects the resolved locale from the response/path and rewrites only when the resolved path is one of the two homepage routes.
- **Component server/client boundaries.** Some components currently rely on `theme` imported directly. `useTranslations` requires the component to be in a context where the locale is known; root layout provides that, so existing server/client boundaries stay unchanged.
- **`navItems.tsx` shape.** Currently a static array. To pick up translations, it becomes either a hook (`useNavItems()`) or a function that takes a `t` argument. Either is straightforward but every consumer (Header, MobileHeader) needs the import path updated.

## Verification

The redesign is verified by:
1. `npm run check` passes with zero lint/typecheck errors.
2. Visiting `/` renders Italian content; visiting `/en` renders English content; both preserve the existing layout, fonts, gradients, and animations.
3. Visiting `/mobile` and `/en/mobile` (or hitting `/` with a mobile UA) renders the mobile pages in the correct locale.
4. The language switcher in the desktop header changes the URL (and the cookie) and reloads the page in the new locale, staying on the same path.
5. No section has visibly mixed languages or untranslated keys (`hero.title` rendered as a literal string, etc.).
