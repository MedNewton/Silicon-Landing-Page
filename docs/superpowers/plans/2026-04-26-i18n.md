# i18n (English + Italian) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Localize the entire site into Italian and English. Italian is the default locale at bare URLs (`/`, `/about/who-are-we`, `/mobile`). English is served under `/en` (`/en`, `/en/about/who-are-we`, `/en/mobile`). The header `LanguageSwitcher` actually navigates between locales while preserving the current pathname.

**Architecture:** Adopt `next-intl` with `localePrefix: "as-needed"`. Move every page under a new `[locale]` segment. Phase 1 lays the infrastructure (middleware, config, JSON catalogs with every key authored, language switcher rewire). Phase 2 swaps hardcoded strings to `t()` / `t.rich()` calls one section at a time. Phase 3 is final verification.

**Tech Stack:** Next.js 15 App Router · React 19 · TypeScript · MUI 7 · `next-intl` (new) · existing UA-mobile middleware (preserved & chained).

**Verification model:** No unit tests — codebase has no test framework. Verification is `npm run check` + `npm run build` + manual smoke at `/`, `/en`, `/mobile`, `/en/mobile`, `/about/*` and `/en/about/*` plus a language-switcher round-trip.

**Spec:** `docs/superpowers/specs/2026-04-25-i18n-design.md`

**Plan amendment vs. spec:** the spec was written before the four About pages existed (`/about/who-are-we`, `/about/vision`, `/about/mission`, `/about/values`). This plan adds them to Phase 2 and to the message catalogs. No changes to spec architecture.

## Conventions

- **Server vs. client components.** Existing client components (`"use client"`) use `useTranslations` from `next-intl`. Existing server components use the async `getTranslations` from `next-intl/server`. We never change a component's client/server status as part of i18n.
- **Import paths.** All locale-aware navigation (`Link`, `useRouter`, `usePathname`, `redirect`) imports from `@/i18n/navigation`, never from `next/link` or `next/navigation` directly. This is what makes locale-prefix preservation automatic.
- **Translation keys.** Top-level namespace per component or section: `header`, `hero`, `cta`, `bento`, `comparison`, `roadmap`, `howItWorks`, `consultants`, `testimonials`, `aiDocumentGeneration`, `pricing`, `footer`, `whoAreWe`, `vision`, `mission`, `values`, `languageSwitcher`. Within a namespace, kebab-no, camelCase keys.
- **Arrays.** Card lists, testimonial rows, pricing tiers, etc. live as JSON arrays under the relevant namespace. Components consume them via `t.raw("namespace.array")` and iterate.
- **Rich text.** Inline HTML-ish formatting (the `<bold>800</bold>` in the hero badge, the `<strike>complex</strike>` in the comparison title) uses `t.rich("key", { bold: (chunks) => <span...>{chunks}</span> })`.
- **Italian translations.** Where the plan provides English text, the implementer authors the Italian equivalent in the same commit. Italian translations are the implementer's responsibility — they are part of the implementation, not "TODO" placeholders. Where idiom matters (taglines, marketing copy), favor a natural, founder-targeted Italian phrasing over a literal translation.

const en = require('./messages/en.json');
const it = require('./messages/it.json');
function keys(obj, prefix = '') {
  const out = [];
  for (const [k, v] of Object.entries(obj)) {
    const path = prefix ? prefix + '.' + k : k;
    if (Array.isArray(v)) {
      v.forEach((item, i) => {
        if (item && typeof item === 'object') out.push(...keys(item, path + '[' + i + ']'));
        else out.push(path + '[' + i + ']');
      });
    } else if (v && typeof v === 'object') {
      out.push(...keys(v, path));
    } else {
      out.push(path);
    }
  }
  return out;
}
const enKeys = keys(en).sort();
const itKeys = keys(it).sort();
const missing = enKeys.filter(k => !itKeys.includes(k));
const extra = itKeys.filter(k => !enKeys.includes(k));
if (missing.length || extra.length) {
  console.error('MISSING in IT:', missing);
  console.error('EXTRA in IT:', extra);
  process.exit(1);
}
console.log('Key shapes match. Total keys:', enKeys.length);

```ts
import { getRequestConfig } from "next-intl/server";
import { hasLocale } from "next-intl";
import { routing } from "@/i18n/routing";

export default getRequestConfig(async ({ requestLocale }) => {
    const requested = await requestLocale;
    const locale = hasLocale(routing.locales, requested)
        ? requested
        : routing.defaultLocale;

    return {
        locale,
        messages: (await import(`../../messages/${locale}.json`)).default,
    };
});
```

## Task 6: Rewrite the middleware to chain next-intl + mobile-UA

**Files:**
- Modify: `src/middleware.ts`

The current middleware rewrites `/` → `/mobile` (and `/team` → `/team/mobile`) for mobile/tablet user agents. With locales added, the rewrite must operate per-locale: `/` → `/mobile` for IT, `/en` → `/en/mobile` for EN.

- [ ] **Step 1: Replace `src/middleware.ts`**

```ts
import { NextResponse, type NextRequest, userAgent } from "next/server";
import createIntlMiddleware from "next-intl/middleware";
import { routing } from "@/i18n/routing";

const intlMiddleware = createIntlMiddleware(routing);

export function middleware(req: NextRequest): NextResponse {
    const { pathname } = req.nextUrl;
    const ua = userAgent(req);
    const deviceType = ua.device.type ?? "desktop";
    const isMobile = deviceType === "mobile" || deviceType === "tablet";

    // Locale-aware homepage paths that should rewrite to a mobile variant
    // when the requesting device is mobile/tablet.
    const mobileHomepagePaths = new Set(["/", "/en"]);

    if (isMobile && mobileHomepagePaths.has(pathname)) {
        const url = req.nextUrl.clone();
        url.pathname = pathname === "/" ? "/mobile" : `${pathname}/mobile`;
        return NextResponse.rewrite(url);
    }

    return intlMiddleware(req);
}

export const config = {
    // Skip api, _next, and static assets. Match everything else, including
    // the bare / and /en plus all locale-prefixed routes.
    matcher: ["/((?!api|_next|_vercel|.*\\..*).*)"],
};
```

**Notes:**
- The `mobileHomepagePaths` set replaces the old hardcoded `["/", "/team"]` list. We drop `/team` entirely — it never had a real route.
- Mobile rewrite runs **before** `intlMiddleware` because the rewrite changes `pathname` to `/mobile` or `/en/mobile`, and we want the rewritten URL to flow through next-intl's locale resolution. (next-intl handles `/en/mobile` correctly: it strips the `/en` prefix internally and resolves to the EN locale.)
- The matcher pattern excludes static asset and API paths, mirroring next-intl's default matcher.

## Phase 2 — String replacement

Each task in Phase 2 swaps hardcoded strings in one section / surface for `t()` calls and ships in its own commit. The pattern is the same every time:

1. Decide whether the file is a server or client component (don't change it).
2. Add the appropriate import:
   - Client: `import { useTranslations } from "next-intl";` and `const t = useTranslations("namespace");`
   - Server: `import { getTranslations } from "next-intl/server";` and `const t = await getTranslations("namespace");`
3. Replace each hardcoded string with `t("key")` or `t.rich("key", { ...formatters })`.
4. For arrays (testimonials, pricing tiers, etc.): `const items = t.raw("namespace.array") as Array<{...}>;`.
5. Run `npm run check`.
6. Commit.

I show one full worked example below (Task 9 — Header). Subsequent tasks (10–24) follow the same pattern; the keys are listed but the JSX wiring is left to the implementer because the surrounding component-specific logic differs case-by-case.
